# AGENTS.md — Durable project context for every session

---

## 1. Project

A simplified X/Twitter clone built with enterprise-grade practices at intentionally small scale (~5 concurrent users for MVP). Users can sign up, post short messages (≤255 chars), follow other users, comment on posts and comments, and like posts. Unauthenticated visitors can read the public feed. The project is feature-scoped as an MVP but held to production standards: strict TypeScript, full migrations, automated tests alongside every feature, CI on every push, and CD gated on passing tests.

## 3. Hard Constraints

- **TypeScript strict; `any` is never used in application code.** Disable the rule explicitly if a third-party boundary forces it, and leave a comment explaining why.
- **ALL client-server communication goes through tRPC** — queries, mutations, and subscriptions. Server Actions are intentionally NOT used. There is one pattern for everything.
- **Better Auth owns its auth tables.** `auth-schema.ts` is generated by `npx auth@latest generate` and must never be hand-edited. Custom user fields (`displayName`, `bio`, `avatarUrl`, etc.) are declared via `additionalFields` in `auth.ts`; the CLI picks them up on next generate. `drizzle-kit` is the **sole migration owner** — it runs `generate` and `migrate`; Better Auth's CLI only emits TypeScript, never SQL.
- **Tests are written alongside features, never deferred.** A feature is not done until its golden-path and key edge-case tests pass.
- **The `user` table gains no new columns until Phase 3.** `displayName`, `bio`, and `avatarUrl` are Phase 3 additions declared via `additionalFields` in `auth.ts`. Do not add them to `auth.ts` or regenerate `auth-schema.ts` before Phase 3 begins. The Phase 1 and Phase 2 `user` table contains only the columns Better Auth emits by default for the `username` plugin.

## 4. Database Schema

### Auth tables — `src/db/auth-schema.ts` (DO NOT HAND-EDIT)
> Generated by `npx auth@latest generate`. Re-run when auth config or plugins change.

| Table | Key columns |
|---|---|
| `user` | `id`, `username` _(unique)_, `email` _(unique)_, `emailVerified`, `displayName`, `bio`, `avatarUrl`, `createdAt`, `updatedAt` |
| `session` | `id`, `token` _(unique, indexed)_, `userId` → `user.id`, `expiresAt` |
| `account` | `id`, `userId` → `user.id`, `providerId`, `accountId`, `password` |
| `verification` | `id`, `identifier`, `value`, `expiresAt` |

### Application tables — `src/db/schema.ts` (our code)

| Table | Key columns | Relations |
|---|---|---|
| `posts` | `id`, `content` (≤255), `authorId` → `user.id`, `createdAt` | author, comments, likes |
| `comments` | `id`, `content`, `authorId` → `user.id`, `postId` → `posts.id`, `parentCommentId` → `comments.id` (nullable) | author, post, parent, children |
| `likes` | `id`, `userId` → `user.id`, `postId` → `posts.id` | unique(userId, postId) |
| `follows` | `id`, `followerId` → `user.id`, `followingId` → `user.id` | unique(followerId, followingId) |

## 6. Migration Workflow

### Initial setup or after any schema change

```bash
# 1. Only when auth config / plugins change:
npx auth@latest generate --output src/db/auth-schema.ts

# 2. Always after any schema file change:
npx drizzle-kit generate

# 3. Apply to database:
npx drizzle-kit migrate
```

### Adding a Better Auth plugin later

```bash
npx auth@latest generate --output src/db/auth-schema.ts   # rewrites the file; CLI prompts before overwrite
npx drizzle-kit generate                                    # diffs against previous snapshot → new SQL migration
npx drizzle-kit migrate
```

**Rules:**
- `drizzle-kit push` is for local prototyping only — never used in CI or production.
- `npx auth migrate` does not work with Drizzle (the CLI explicitly errors out). Always go through `drizzle-kit`.
- Migration SQL files are committed to source control and reviewed like any other code change.

### ⚠️ Verify commands and versions before use

The version numbers and CLI invocations in this file reflect what was researched at planning time. Before running any command, verify:

```bash
# Check the current latest versions at build time — do not trust hardcoded numbers in this file
npm show better-auth version          # check Better Auth current stable
npm show drizzle-orm version          # check Drizzle ORM current stable
npm show drizzle-kit version          # check drizzle-kit current stable
npm show @trpc/server version         # check tRPC current stable
npx auth@latest --version             # confirm Better Auth CLI resolves correctly
```

Always run `npx auth@latest` (not a pinned version) so the CLI matches the installed `better-auth` package. If any tool's major version has changed from what is documented here, check the official changelog for breaking changes before proceeding.

## 7. Phases

### ✅ Phase 1 — Foundation · **CURRENT FOCUS**

> Thinnest deployable slice: a real app where someone can sign up, read posts, and write one.

**Deliverables checklist:**
- [ ] Next.js 16 project with TypeScript strict, ESLint + Prettier + `@typescript-eslint`, Tailwind CSS v4
- [ ] Drizzle ORM configured; `auth-schema.ts` generated; `posts` table; initial migration applied
- [ ] Better Auth wired up: sign up, sign in, sign out (no email verification)
- [ ] tRPC router: `feed.getPublic` (cursor-paginated query) + `post.create` (auth-guarded mutation)
- [ ] Pages: `/` (public feed), `/login`, `/register`, `/compose` (auth-guarded)
- [ ] Sentry configured (`@sentry/nextjs`, source map upload in build)
- [ ] Dependabot (`.github/dependabot.yml`) + Socket.dev GitHub App installed
- [ ] GitHub Actions CI: lint → typecheck → Vitest → Playwright
- [ ] Railway project created: app service + Postgres service + "Wait for CI" enabled
- [ ] PR environments enabled on Railway
- [ ] Vitest: post `content` validation unit tests
- [ ] Playwright: sign-up golden path E2E, create-post golden path E2E
- [ ] Deployed and accessible at a Railway URL

**Phase 1 explicitly excludes:**
- Comments, likes, follows
- Home feed (`/home`) and profile pages (`/[username]`)
- SSE / real-time updates
- Profile settings, password/email change
- Avatar uploads and Cloudflare R2
- Nested comments
- `post.delete`, `user.*` procedures

---

### Phase 2 — Social Layer

- DB: `comments`, `likes`, `follows` tables + migrations
- tRPC: `comment.create/list/delete`, `like.toggle`, `user.getProfile/follow/unfollow`, `feed.getHome`
- Pages: `/[username]` (public profile), `/home` (authenticated feed), `/[username]/[postId]` (post + thread)
- Like counts visible to all (including unauthenticated)
- Tests: follow/unfollow golden path, like toggle, nested comment thread, home feed isolation edge case, unauthenticated like rejection edge case

---

### Phase 3 — Real-time + Profile Management

- tRPC subscription: `feed.onNewPost` (SSE with `lastEventId` reconnect catch-up, in-memory EventEmitter)
- DB: `avatarUrl`, `bio`, `displayName` fields on `user` (via `additionalFields` → regenerate `auth-schema.ts`)
- tRPC: `user.updateProfile`, `user.changePassword`, `user.changeEmail`, `storage.getPresignedUploadUrl`
- R2 integration: presigned upload URL → client uploads directly → URL stored in Postgres
- Pages: `/[username]/settings`
- Profile page: user's own posts + all comments on their posts (including nested)
- Tests: SSE reconnect integration test, profile update golden path, avatar upload golden path

---

### Phase 4 — Edge Cases + Observability Polish

- Playwright edge cases: auth redirect on session expiry, empty feed/profile states, 255-char limit enforcement in UI, cross-user feed isolation, follow/unfollow idempotency
- Sentry: validate session replay capturing on errors, tune `tracesSampleRate` and `replaysOnErrorSampleRate` to stay within free tier
- Final CI hardening: Vitest coverage thresholds, Playwright retry config, `socket ci` confirmed in pipeline
